# Spark Parquet Schema Reading In Depth

## 1. Overview

When reading Parquet files, Spark must first determine the schema of the data (column names, column types, nullability, etc.). How the schema is resolved directly affects both read performance and correctness.

Spark supports three ways to obtain the schema for Parquet data:

| Mode | Files Read | Performance | Use Case |
|------|-----------|-------------|----------|
| User-specified at table creation | 0 | Fastest | Schema is known; maximum performance desired |
| Automatic inference (default) | 1 | Fast | Schema is stable and consistent across files |
| Merge inference (`mergeSchema=true`) | All | Slow | Schema evolves over time |

## 2. Three Schema Resolution Modes

### 2.1 User-Specified Schema

Specifying column definitions in the table creation statement **completely skips** footer reading and inference.

**Create a temporary view with a schema:**

```sql
CREATE TEMPORARY VIEW user_data (name STRING, age INT)
USING parquet
OPTIONS (path '/path/to/data');

SELECT * FROM user_data;
```

**Create an external table with a schema:**

```sql
CREATE TABLE user_data (name STRING, age INT)
USING parquet
LOCATION '/path/to/data';

SELECT * FROM user_data;
```

**Behavior:**
- No file footers are read; best performance
- The `mergeSchema` option is ignored
- Columns specified but absent from the file are returned as `null`
- Columns present in the file but not in the schema are skipped (column pruning)

### 2.2 Automatic Inference (Default)

When no schema is specified, Spark reads schema information from the Parquet file footer. By default it reads only **one** file, assuming all files share the same schema.

```sql
-- Query Parquet files directly (schema inferred automatically)
SELECT * FROM parquet.`/path/to/data`;

-- Or create a view without column definitions (triggers inference)
CREATE TEMPORARY VIEW user_data
USING parquet
OPTIONS (path '/path/to/data');

SELECT * FROM user_data;
```

### 2.3 Merge Inference

With `mergeSchema` enabled, Spark reads the schemas from **all** data files and merges them into a superset.

```sql
-- Option 1: Specify via OPTIONS
CREATE TEMPORARY VIEW user_data
USING parquet
OPTIONS (path '/path/to/data', mergeSchema 'true');

SELECT * FROM user_data;

-- Option 2: Enable via global configuration
SET spark.sql.parquet.mergeSchema = true;
SELECT * FROM parquet.`/path/to/data`;
```

## 3. Schema Inference In Detail

### 3.1 Schema Information Inside Parquet Files

The footer of every Parquet file contains two sets of schema information:

1. **Parquet native schema** (`MessageType`): The Parquet format's own type system. Always present.
2. **Spark SQL schema** (JSON string): Embedded by Spark in the file's key-value metadata (key: `org.apache.spark.sql.parquet.row.metadata`). Only present in files written by Spark.

Additionally, a directory may contain two kinds of summary files:

| File Name | Description |
|-----------|-------------|
| `_common_metadata` | Contains only the merged schema information; small in size |
| `_metadata` | Contains the merged schema plus all row group information; can be large |

> Note: Summary files are no longer generated by default in recent versions of Spark (`parquet.summary.metadata.level=NONE`). They primarily exist in data written by older versions.

### 3.2 File Selection Strategy

Before inference, Spark classifies files in the directory into three categories, then applies a different strategy depending on whether merging is enabled:

**Without merging (default) -- only one file is read:**

```
Priority: _common_metadata > _metadata > first data file (by lexicographic path order)
```

Only the footer of the highest-priority file is read.

**With merging -- all files are read:**

```
Files read: all data files + _metadata + _common_metadata
```

When `spark.sql.parquet.respectSummaryFiles=true`, Spark trusts the schema in the summary files and skips reading data file footers entirely.

### 3.3 Schema Extraction Priority

When extracting a schema from a single file, Spark preferentially uses the Spark SQL schema:

1. Attempt to read the Spark SQL schema JSON string from the key-value metadata
2. Attempt to parse it as JSON (current format)
3. Fall back to the legacy `DataType.fromCaseClassString` parser
4. If all parsing fails, fall back to converting the Parquet native `MessageType` to a Spark `StructType`

Using the Spark SQL schema preserves Spark-specific type precision (e.g., `ByteType`, `ShortType`), since Parquet only has `INT32` and cannot distinguish between these types.

### 3.4 Parquet-to-Spark Type Mapping

When converting from the Parquet native schema, the following mapping rules apply:

**Primitive types:**

| Parquet Physical Type | Logical Annotation | Spark Type |
|-----------------------|-------------------|------------|
| BOOLEAN | — | BooleanType |
| FLOAT | — | FloatType |
| DOUBLE | — | DoubleType |
| INT32 | none | IntegerType |
| INT32 | INT(8, signed) | ByteType |
| INT32 | INT(16, signed) | ShortType |
| INT32 | DATE | DateType |
| INT32 | DECIMAL(p,s) | DecimalType(p,s), p <= 9 |
| INT64 | none | LongType |
| INT64 | DECIMAL(p,s) | DecimalType(p,s), p <= 18 |
| INT64 | TIMESTAMP(MICROS, adjusted=true) | TimestampType |
| INT64 | TIMESTAMP(MICROS, adjusted=false) | TimestampNTZType (requires config) |
| INT64 | TIME(MICROS) | TimeType |
| INT96 | — | TimestampType (requires `int96AsTimestamp=true`) |
| BINARY | UTF8 / ENUM / JSON | StringType |
| BINARY | none | BinaryType (or StringType if `binaryAsString` is set) |
| BINARY | DECIMAL(p,s) | DecimalType(p,s) |
| FIXED_LEN_BYTE_ARRAY | DECIMAL(p,s) | DecimalType(p,s) |

**Complex types:**

| Parquet Structure | Spark Type |
|-------------------|------------|
| GROUP + LIST annotation | ArrayType |
| GROUP + MAP annotation | MapType |
| GROUP (no annotation) | StructType |
| GROUP + VARIANT annotation | VariantType |

**Configurations that affect type inference:**

| Configuration | Default | Effect |
|---------------|---------|--------|
| `spark.sql.parquet.binaryAsString` | false | When true, unannotated BINARY is inferred as StringType |
| `spark.sql.parquet.int96AsTimestamp` | true | When true, INT96 is inferred as TimestampType |
| `spark.sql.parquet.inferTimestampNTZ.enabled` | true | When true, timestamps without `adjustedToUTC` are inferred as TimestampNTZType |

## 4. Schema Merging In Detail

### 4.1 When It Triggers

Schema merging is triggered when either of the following conditions is met:

```sql
-- Option 1: Specify via data source option (takes priority over global config)
CREATE TEMPORARY VIEW user_data
USING parquet
OPTIONS (path '/path/to/data', mergeSchema 'true');

-- Option 2: Global configuration
SET spark.sql.parquet.mergeSchema = true;
```

The default is `false` (merging disabled).

Note: If column definitions (schema) are explicitly specified at table creation, merging will not occur even with `mergeSchema=true` -- the user-specified schema always takes the highest priority.

### 4.2 Merge Workflow

The merge leverages Spark's distributed computing capabilities in two rounds:

```
          File list [f1, f2, f3, ..., fn]
                        |
                  parallelize
                        |
        +---------------+---------------+
        v               v               v
   Partition 1     Partition 2     Partition 3
   +----------+   +----------+   +----------+
   |read footer|   |read footer|   |read footer|
   |extract    |   |extract    |   |extract    |
   | schema    |   | schema    |   | schema    |
   |local merge|   |local merge|   |local merge|
   +-----+----+   +-----+----+   +-----+----+
         |               |               |
         +-------+-------+-------+-------+
                 v               v
            collect() to Driver
                 |
                 v
          Final merge on Driver
                 |
                 v
           Final StructType
```

- Parallelism = `min(number of files, spark.default.parallelism)`
- Footer reads use 8 threads in parallel and skip row group information (`SKIP_ROW_GROUPS`), reading only metadata
- Partial merges happen on Executors; the final merge happens on the Driver

### 4.3 Merge Algorithm

The core of merging is the `StructType.merge` method -- a recursive, field-name-based union algorithm.

**Struct-level field merging:**

```
merge({A: T1, B: T2}, {B: T3, C: T4})
-> {A: T1, B: merge(T2, T3), C: T4}
```

Rules:
1. **Fields with the same name**: Recursively merge their types; nullability is OR-ed
2. **Fields only in the left schema**: Retained as-is
3. **Fields only in the right schema**: Appended to the end of the result

**Recursive type merge rules:**

| Type Combination | Merge Rule |
|-----------------|------------|
| StructType + StructType | Recursively merge by field name (as above) |
| ArrayType + ArrayType | Recursively merge element types; `containsNull` is OR-ed |
| MapType + MapType | Recursively merge key and value types independently; `valueContainsNull` is OR-ed |
| DecimalType + DecimalType | Scale must be identical; precision takes the max |
| Identical types | Returned directly |
| Other incompatible types | **Exception thrown** |

**Field ordering:** Files are sorted by path in lexicographic order. The field order from the first file determines the positions of those fields in the result; new fields from subsequent files are appended at the end.

### 4.4 Handling Heterogeneous Schemas

Below is a complete SQL example that illustrates the various heterogeneous schema scenarios.

**Prepare data:** Assume two batches of data are written into different partitions of the same directory:

```sql
-- First batch: only name and age
CREATE TABLE batch1 USING parquet LOCATION '/data/users/batch=1'
  AS SELECT 'Alice' AS name, 30 AS age;

-- Second batch: adds an email column
CREATE TABLE batch2 USING parquet LOCATION '/data/users/batch=2'
  AS SELECT 'Bob' AS name, 25 AS age, 'bob@test.com' AS email;
```

**Without merging, only the first file's schema is visible:**

```sql
SELECT * FROM parquet.`/data/users`;
-- Result has only name, age columns (schema from the first file)
-- The email column from the second batch is discarded
```

**With merging enabled, the union of all columns is returned:**

```sql
SET spark.sql.parquet.mergeSchema = true;
SELECT * FROM parquet.`/data/users`;
-- +-----+---+------------+-----+
-- | name|age|       email|batch|
-- +-----+---+------------+-----+
-- |Alice| 30|        null|    1|  <- email is missing in batch 1, filled with null
-- |  Bob| 25|bob@test.com|    2|
-- +-----+---+------------+-----+
```

The following subsections describe each heterogeneous scenario in detail.

#### 4.4.1 Different Number of Columns -- Normal Merge

```
file1: {name: STRING, age: INT}
file2: {name: STRING, age: INT, email: STRING}

Merged result: {name: STRING, age: INT, email: STRING}
Reading file1: email column values are null
```

```sql
-- Rows from file1 have email = null
SET spark.sql.parquet.mergeSchema = true;
SELECT name, age, email FROM parquet.`/path/to/data`;
```

#### 4.4.2 Different Column Names -- Union

```
file1: {x: INT, y: INT}
file2: {y: INT, z: INT}

Merged result: {x: INT, y: INT, z: INT}
Reading file1: z = null
Reading file2: x = null
```

#### 4.4.3 Different Nested Struct Fields -- Recursive Merge

```
file1: {info: {city: STRING}}
file2: {info: {city: STRING, zip: INT}}

Merged result: {info: {city: STRING, zip: INT}}
Reading file1: info.zip = null
```

```sql
SET spark.sql.parquet.mergeSchema = true;
SELECT info.city, info.zip FROM parquet.`/path/to/data`;
-- Rows from file1: city has a value, zip = null
-- Rows from file2: both city and zip have values
```

#### 4.4.4 Different Structs Inside Arrays -- Recursive Element Type Merge

```
file1: {tags: Array<{k: STRING}>}
file2: {tags: Array<{k: STRING, v: INT}>}

Merged result: {tags: Array<{k: STRING, v: INT}>}
```

#### 4.4.5 Different Map Value Types -- Recursive Value Type Merge

```
file1: {props: Map<STRING, {a: INT}>}
file2: {props: Map<STRING, {a: INT, b: STRING}>}

Merged result: {props: Map<STRING, {a: INT, b: STRING}>}
```

#### 4.4.6 Different Decimal Precision -- Takes the Larger Precision

```
file1: {amount: DECIMAL(10, 2)}
file2: {amount: DECIMAL(15, 2)}

Merged result: {amount: DECIMAL(15, 2)}
```

Note: The scale (number of decimal places) must be identical; otherwise an exception is thrown.

```sql
-- Decimal precision is automatically set to max
SET spark.sql.parquet.mergeSchema = true;
SELECT amount FROM parquet.`/path/to/data`;
-- Result column type is DECIMAL(15, 2)
```

#### 4.4.7 Incompatible Types -- Exception Thrown

The following cases throw a `CANNOT_MERGE_SCHEMAS` exception:

```sql
-- Scenario 1: Same column name but different physical types (LONG vs INT); no implicit promotion
-- file1 has id as BIGINT, file2 has id as INT
SET spark.sql.parquet.mergeSchema = true;
SELECT * FROM parquet.`/path/to/data`;
-- Error: [CANNOT_MERGE_SCHEMAS] Failed merging schemas...

-- Scenario 2: Completely incompatible types for the same column name (STRING vs INT)
-- Also throws an error

-- Scenario 3: Different Decimal scales (DECIMAL(10,2) vs DECIMAL(10,3))
-- Also throws an error
```

**Important: Spark's schema merging does NOT perform implicit type promotion (e.g., INT -> LONG). Types must match exactly or fall under one of the special mergeable cases described above.**

#### 4.4.8 Nullability Differences -- OR-ed

```
file1: {name: STRING (not null)}
file2: {name: STRING (nullable)}

Merged result: {name: STRING (nullable)}
```

If either side is nullable, the result is nullable.

#### 4.4.9 Case Sensitivity

```
file1: {col: INT}
file2: {COL: INT}
```

```sql
-- Default case-insensitive: treated as the same column
SET spark.sql.caseSensitive = false;
SET spark.sql.parquet.mergeSchema = true;
SELECT * FROM parquet.`/path/to/data`;
-- Result has one column: col (or COL, depending on which file is processed first)

-- Case-sensitive mode: treated as two different columns
SET spark.sql.caseSensitive = true;
SELECT * FROM parquet.`/path/to/data`;
-- Result has two columns: col and COL
```

### 4.5 Read Behavior After Merging

The merged superset schema is passed to each file's reader as the "requested schema". When reading each file, Spark reconciles schema differences through the **Schema Clipping** mechanism:

1. **Columns present in both the file and the schema**: Read normally
2. **Columns in the schema but missing from the file**: Filled with `null`
3. **Columns in the file but not in the schema**: Skipped (column pruning), no performance impact

This mechanism applies to nested types as well -- missing nested fields are also filled with `null`.

## 5. Scenario Behavior Matrix

| Scenario | mergeSchema=false | mergeSchema=true |
|----------|-------------------|-----------------|
| All files share the same schema | Works normally | Works normally (but reads all footers unnecessarily) |
| New files have added columns | Rows from new files may fail or lose data | Merges automatically; old files have null for new columns |
| Files have completely different columns | Uses the first file's schema; other columns are lost | All columns are unioned |
| Same column name with different types (INT vs LONG) | May throw a runtime exception during reading | Throws an exception during the merge phase |
| Different fields within nested structs | Reads using the first file's struct schema | Recursively merges struct fields |
| Different Decimal precision with the same scale | Reads using the first file's precision | Automatically takes the larger precision |
| Column definitions specified at table creation | Reads using the user-specified schema | Reads using the user-specified schema (mergeSchema is ignored) |

## 6. Configuration Reference

### Core Configurations

| Configuration | Default | Description |
|---------------|---------|-------------|
| `spark.sql.parquet.mergeSchema` | false | Whether to enable schema merging |
| `spark.sql.parquet.respectSummaryFiles` | false | When merging, whether to trust summary files and skip reading data file footers |

### Type Inference Configurations

| Configuration | Default | Description |
|---------------|---------|-------------|
| `spark.sql.parquet.binaryAsString` | false | Infer unannotated BINARY as STRING |
| `spark.sql.parquet.int96AsTimestamp` | true | Infer INT96 as Timestamp |
| `spark.sql.parquet.inferTimestampNTZ.enabled` | true | Infer timestamps without adjustedToUTC as TimestampNTZ |
| `spark.sql.parquet.fieldId.read.enabled` | true | Use Parquet field IDs to match columns |

### Data Source Options

| Option | Default | Description |
|--------|---------|-------------|
| `mergeSchema` | Inherits global config value | Overrides the global `spark.sql.parquet.mergeSchema` configuration |

## 7. Performance and Best Practices

### 7.1 Stable Schema: Use the Default Settings

If the data schema does not change, keep `mergeSchema=false` (the default). Spark reads only one file's footer, with minimal overhead.

### 7.2 Known Schema: Specify Explicitly

In production environments where the schema is known, explicitly specifying column definitions at table creation is optimal:

```sql
CREATE TABLE user_data (id BIGINT, name STRING, ts TIMESTAMP)
USING parquet
LOCATION '/path/to/data';
```

Benefits:
- Zero footer-reading overhead
- Avoids ambiguity during inference
- Explicit control over column types

### 7.3 Schema Evolution: Enable Merging As Needed

Only enable `mergeSchema` when you truly need to read files with different schemas. It is recommended to specify this at the data source level rather than changing the global configuration:

```sql
CREATE TEMPORARY VIEW user_data
USING parquet
OPTIONS (path '/path/to/data', mergeSchema 'true');

SELECT * FROM user_data;
```

### 7.4 Large Number of Files: Leverage Summary Files

When a directory contains a large number of Parquet files and schema merging is needed, reading each file's footer individually can be slow (especially on object stores like S3). To optimize:

1. Generate summary files at write time:

   ```sql
   SET parquet.summary.metadata.level = ALL;
   ```

2. Trust summary files at read time:

   ```sql
   SET spark.sql.parquet.respectSummaryFiles = true;
   SET spark.sql.parquet.mergeSchema = true;
   SELECT * FROM parquet.`/path/to/data`;
   ```

This way Spark only reads the summary files, avoiding the need to scan each data file's footer.

### 7.5 Caveats

- Schema merging **does not perform implicit type promotion**. If the same column has different types across files (e.g., INT vs LONG), the merge will fail. Ensure type consistency at write time.
- The column order in the merged result depends on the lexicographic ordering of file paths, which may vary across environments. If column order matters, either specify the schema explicitly or use `SELECT` to reorder columns after reading.
- When reading Parquet files, all columns are marked as nullable by default (regardless of the definition in the file), for compatibility reasons.
